## 1. 仓库主线与统一接口

本轮工作始终遵守以下主线约束：

- 先稳定单条件抓取生成主线
- 先做 `world` frame，再看 `camera`
- 先支持 `global/partial` 点云的统一 train/sim 接口
- 当前仓库统一输出语义保持为：
  - `init_pose`
  - `squeeze_pose`
  - `squeeze_joint`
- 统一算法入口保持在 `BaseSCModel` 体系下，避免为每个生成算法重写 `train_sc.py / sim_sc.py`
- `PointNet / BPS` backbone 保持可替换

## 2. DexDiffuser 迁移设计

### 2.1 第一版迁移边界

最初确定 `DexDiffuser` 的第一版只迁移生成器主链，不迁移：

- evaluator / guidance
- refine.py
- 原始 Hydra / dataset / train/sample 脚本

迁移目标是：

- `models/dexdiffuser/` 放网络主体与扩散核心
- `models/dexdiffuser_sc.py` 作为单条件统一接口壳层
- `DexDiffuserSCModel(BaseSCModel)` 负责对接当前仓库 batch 语义

对应设计文档：

- [20260330_方案_DexDiffuser生成器迁移设计.md](/home/ccs/repositories/dexgrasp_learning/docs/20260330_方案_DexDiffuser生成器迁移设计.md)

### 2.2 表示层决策

没有沿用原始 DexDiffuser 的 `3 + 6 + joint` 输出语义，而是直接适配当前仓库目标：

```text
x = [init_pose(6) | squeeze_pose(6) | squeeze_joint(20)]
Dx = 32
```

原因：

- 当前仓库监督已经统一在 `se3_log(6D)` 语义上
- 当前主线输出不是单个最终 grasp，而是 `init + squeeze + joint`
- 如果为了迁移 DexDiffuser 再把表示改回原始格式，会破坏当前统一接口

因此 `representation.py` 负责：

- `build_diffusion_target(batch) -> [B, 32]`
- `split_diffusion_target(x) -> structured outputs`

### 2.3 条件编码决策

后续进一步明确了一个关键约束：

- 不改 `PointNet` 和 `BPS` 原始 backbone 代码
- token 化全部放在 `models/dexdiffuser/condition.py` 的外部适配层

实现路径：

- `PointNet`：
  - 复用其 `global_feature + local_feature`
  - 在 adapter 中采样局部 token，投影到 `context_dim`
- `BPS`：
  - 不改原始 `BPSEncoder`
  - `DexDiffuser` 自己从点云重新构造算法专用 BPS token

这样做的收益：

- `CVAE` 和 evaluator 不受影响
- `DexDiffuser` 保留 token 条件优势
- backbone 仍然保持算法无关

### 2.4 主接口设计

`DexDiffuserSCModel` 统一提供：

- `forward(batch)`：训练，返回 `loss`
- `sample(batch, num_samples)`：并行生成 `K` 个候选
- `infer(batch)`：`K=1` 的包装

`DDPM` 只处理：

- `compute_loss(x0, context_tokens)`
- `sample(context_tokens, num_samples, sample_shape)`

`UNet` 接口保持：

- `x_t: [B, Dx]`
- `timesteps: [B]`
- `context: [B, T_ctx, C_ctx]`

## 4. DexDiffuser 调参：训练步数是最有效变量

### 4.1 20k -> 30k -> 40k -> 70k

后续对 `DexDiffuser` 做了较系统的 warm-start 步数扩展。最重要发现是：

- 训练步数/退火周期对性能影响非常大
- 比学习率、`l1` loss、简单 normalization 更有效

关键结果：

- `20k train`: `GSR=0.412573`, `OSR=0.973684`
- `30k train`: `GSR=0.495614`, `OSR=0.997076`
- `20k test`: `GSR=0.472078`, `OSR=1.000000`
- `30k test`: `GSR=0.520779`, `OSR=1.000000`

继续 warm-start：

- `40k test`: `GSR=0.581169`
- `50k test`: `GSR=0.599351`
- `60k test`: `GSR=0.611039`
- `70k test`: `GSR=0.613636`

### 4.2 关键判断

这一阶段明确排掉了几个低优先级方向：

- 单纯抬高学习率

因为这些方向的收益不如“多训练 + 拉长余弦退火周期”明显。

## 6. UDGM 初版、问题定位与改进

### 6.1 初版 UDGM

初版 `UDGM` 被实现为：

- 条件 flow 直接建模完整 `32D`
- 对外保持统一接口

主文件：

- [models/udgm_sc.py](/home/ccs/repositories/dexgrasp_learning/models/udgm_sc.py)
- [models/udgm/](/home/ccs/repositories/dexgrasp_learning/models/udgm)

但后续实验表明：

- 初版 `UDGM` 的性能明显不如 `DexDiffuser`
- 即使训练到 `50k`，也只是到 `GSR≈0.465` 量级

### 6.2 与 DexLearn nflow 的差异定位

进一步分析发现，初版 `UDGM` 和 `DexLearn nflow` 的差距主要在 flow 本体：

- 缺少 `ActNorm`
- 缺少可逆线性混合
- conditioner 太弱
- `loss_clamp_max=20` 对当前 `32D` 语义不合适

对应分析文档：

- [20260401_过程_UDGM与DexLearn_nflow差异及改进.md](/home/ccs/repositories/dexgrasp_learning/docs/20260401_过程_UDGM与DexLearn_nflow差异及改进.md)

已做改进：

- 增加 [models/udgm/layers.py](/home/ccs/repositories/dexgrasp_learning/models/udgm/layers.py)
- 在 flow 中加入 `ActNorm -> InvertibleLinear -> Coupling`
- `coupling / condition` 支持残差网络
- 原始 `udgm` 的 `loss_clamp_max` 提高到 `80`
- 增加 `raw_nll` 与 `clip_fraction` 日志

### 6.3 为什么原始 `UDGM` 会出现 `loss=20`

一度发现 `UDGM` 训练中 `loss_nll` 恒为 `20`。原因不是模型本身，而是：

- `loss_nll = -log_prob`
- 又做了 `clamp_max(20.0)`
- 当前完整 `32D` 目标下，整批 raw NLL 经常 `> 20`
- 导致整个 batch 基本都被 clamp 顶死

这进一步说明：

- `DexLearn` 原始 `flow_thresh=20` 不能直接照搬到当前 `UDGM`
- 因为目标空间、分解方式、loss 组成完全不同

## 7. UDGM_RT：关键改动是训练时不用 GT squeeze_pose 做回归

### 7.1 第一版 teacher forcing 失败

为了模仿 `DexLearn FlowRTJ_MLPRTJ`，后续实现了 `udgm_rt`：

- `flow` 负责 `squeeze_pose`
- 回归头补 `init_pose + squeeze_joint`

但第一版训练时回归头仍然吃 GT `squeeze_pose`，结果是：

- 训练 loss 很好看
- `test sim` 极差

根因是典型的 teacher forcing gap：

- train: `GT squeeze_pose -> regression head`
- test: `sampled squeeze_pose -> regression head`

### 7.2 改成 sampled squeeze_pose 后显著提升

后续明确改成：

- `loss_flow` 仍然用 GT `squeeze_pose`
- 回归头训练时不再用 GT
- 而是从 flow 中采样 `squeeze_pose`，选高 `log_prob` 候选送入回归头

实现文件：

- [models/udgm_rt_sc.py](/home/ccs/repositories/dexgrasp_learning/models/udgm_rt_sc.py)
- [models/udgm/rt.py](/home/ccs/repositories/dexgrasp_learning/models/udgm/rt.py)

这次改动带来了决定性的性能提升：

- 新 `udgm_rt_50k test`: `GSR=0.583766`, `OSR=1.000000`

相比旧版 teacher forcing `udgm_rt` 有数量级提升，并且已经非常接近 `DexDiffuser_50k`：

- `DexDiffuser_50k test`: `GSR=0.599351`, `OSR=1.000000`

阶段性结论：

- `UDGM` 这条线真正有效的不是单纯加强 flow
- 而是训练链路必须与推理链路对齐

## 8. DexDiffuser_RT：为什么没有带来提升

在 `DexDiffuser` 上也试了类似的 `RT` 分解：

- `dexdiffuser_rt`

设计是：

- diffusion 先生成 `squeeze_pose`
- 第二阶段再预测 `init_pose + squeeze_joint`

但结果没有像 `dp_rt` 或 `udgm_rt` 那样明显提升。主要原因：

- 它底层仍然是 `100-step + token UNet + cross-attention`
- 速度主瓶颈没变
- 原版 `DexDiffuser` 本来就有足够能力直接联合建模 `32D`
- 拆开以后，反而让第二阶段失去了原版最有价值的 token 条件

因此：

- `dexdiffuser_rt` 不适合作为主线替代 `dexdiffuser`
- `DexDiffuser` 更适合保留原本的联合建模能力

## 9. DexLearn 风格轻量 diffusion：DP 与 DP_RT

### 9.1 为什么 DexLearn diffusion 很快

一个关键问题是：为什么 `DexLearn` 的 diffusion 采样很快，而 `DexDiffuser` 很慢？

结论是：

- 快的是 `DiffusionRT_MLPRTJ`
- 不是 full diffusion `DiffusionRTJ`

原因：

- 只对一部分变量做 diffusion
- 默认 inference step 很少
- 去噪器是轻量 MLP，不是 token UNet
- 条件只用 global feature

因此实现了新的家族：

- `dp`
- `dp_rt`

主文件：

- [models/dp/](/home/ccs/repositories/dexgrasp_learning/models/dp)
- [models/dp_sc.py](/home/ccs/repositories/dexgrasp_learning/models/dp_sc.py)
- [models/dp_rt_sc.py](/home/ccs/repositories/dexgrasp_learning/models/dp_rt_sc.py)

### 9.2 DP 家族结果

在 `50k / pointnet / world / partial / K=20 / test` 下：

- `dp`: `GSR=0.387013`, `OSR=0.987013`
- `dp_rt`: `GSR=0.519481`, `OSR=0.974026`

结论：

- `dp_rt` 明显优于 `dp`
- 对轻量 diffusion 来说，把任务分解成“先 pose 再补全”确实有效

## 12. grad_clip 来源与 5k probe 结论

### 12.1 来源

`train_sc.py` 中的：

```python
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=grad_clip)
```

不是最近为了某个算法临时加的，而是随统一单条件训练入口在 commit `13df198` 引入的。  
不过 `DexLearn` 的 `nflow` 和 `diffusion` 参考实现也都使用 `grad_clip=10.0`。

### 12.2 实测结果

对 `CVAE` 和 `DexDiffuser` 做了完整 `5000` step probe，结果文件：

- [cvae_clip_10_0_5000.json](/home/ccs/repositories/dexgrasp_learning/outputs/grad_clip_probe/cvae_clip_10_0_5000.json)
- [cvae_clip_1_0_5000.json](/home/ccs/repositories/dexgrasp_learning/outputs/grad_clip_probe/cvae_clip_1_0_5000.json)
- [dexdiffuser_clip_10_0_5000.json](/home/ccs/repositories/dexgrasp_learning/outputs/grad_clip_probe/dexdiffuser_clip_10_0_5000.json)
- [dexdiffuser_clip_1_0_5000.json](/home/ccs/repositories/dexgrasp_learning/outputs/grad_clip_probe/dexdiffuser_clip_1_0_5000.json)

核心统计：

- `CVAE, clip=10`
  - `max_preclip_grad_norm=3.364`
  - `p99=0.886`
  - `clip_fraction=0.0`
- `CVAE, clip=1`
  - `clip_fraction=0.0072`
  - `final_loss` 相比 `clip=10` 降低约 `1.63%`

- `DexDiffuser, clip=10`
  - `max_preclip_grad_norm=3.673`
  - `p99=0.650`
  - `clip_fraction=0.0`
- `DexDiffuser, clip=1`
  - `clip_fraction=0.0034`
  - `final_loss` 相比 `clip=10` 增加约 `2.02%`

对 `DP` 的 probe 跑到约 `2000` step 时，梯度范数仍只有：

- `0.09 ~ 0.12`

说明 `DP` 比 `CVAE / DexDiffuser` 还更不依赖 clip。由于信息已经足够，这条没有继续跑满 `5000`。

### 12.3 结论

- `grad_clip=10` 在当前这些算法上基本只是保险丝
- 不是决定性超参数
- 把阈值收紧到 `1` 后，`loss` 变化也很小
- 真正更值得优先调的是：
  - `DexDiffuser`：训练步数/退火周期
  - `UDGM_RT`：训练链与推理链对齐
  - `DP_RT`：结构分解与轻量采样

## 13. 当前阶段结论

### 13.1 最稳的高成功率主线

- `DexDiffuser` 仍然是当前 accuracy 最强的主线候选
- best `test GSR` 已到 `0.613636`

### 13.2 最有价值的速度/效果折中

- 新版 `UDGM_RT` 已经接近 `DexDiffuser_50k`
- 但采样速度快约 `55x`

### 13.3 轻量 diffusion 的有效方向

- `DP_RT` 明显优于 `DP`
- 说明“先预测更核心变量，再补全其余变量”对轻量 diffusion 很有效

### 13.4 不建议继续优先投入的方向

- 继续强化老版 `UDGM`
- 指望 `dexdiffuser_rt` 替代原版 `dexdiffuser`
